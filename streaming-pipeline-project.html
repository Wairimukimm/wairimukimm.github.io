Problem Statement
The Challenge
Batch ETL pipelines that run hourly or daily are blind to what's happening right now. For use cases like fraud detection, live dashboards, and operational alerting, a decision made on 6-hour-old data is often a decision made too late. The goal was to build a pipeline that processes events as they happen — with correctness guarantees, fault tolerance, and a queryable output layer — not a fragile demo that breaks under real load.
System Design
Architecture

Events flow from producers through a durable log, get processed in-flight with windowed aggregations, and land in two destinations: a low-latency cache for live reads and a warehouse for historical analysis.

Build Process
1

Event Schema & Producer Design

Defined a versioned Avro schema for event payloads to prevent downstream breakage as the event structure evolved. Built lightweight producers simulating realistic traffic patterns — user actions, transaction events, sensor pings — with configurable burst and out-of-order delivery to stress-test the pipeline.

2

Durable Event Log

Set up Kafka as the central nervous system — partitioned by event key to preserve ordering per entity, replicated across brokers for fault tolerance. Configured retention policies balancing replay capability against storage cost, and consumer groups to allow independent scaling of downstream processors.

Apache KafkaPartitioning StrategyConsumer Groups
3

Stream Processing Layer

Built stateful Flink jobs for windowed aggregations (tumbling and sliding windows), deduplication, and enrichment by joining the event stream against a slower-changing reference dataset. Used watermarks to handle out-of-order and late-arriving events without stalling the pipeline indefinitely.

Apache FlinkWindowed AggregationWatermarkingStream-Table Join
4

Serving Layer

Split output into two paths. Aggregated metrics needing sub-second reads (live counters, rolling averages) write to Redis with TTL-based expiry. Full event history and slower analytical queries land in Postgres via a Flink-to-JDBC sink, partitioned by event date for query performance.

RedisPostgresJDBC SinkTTL Caching
5

Fault Tolerance & Exactly-Once Processing

Enabled Flink checkpointing to a durable state backend, allowing the job to recover from failure without reprocessing or dropping events. Configured Kafka transactional producers and idempotent sinks to guarantee exactly-once semantics end to end — critical for financial and counting use cases.

CheckpointingExactly-Once SemanticsState Backend
6

Monitoring & Observability

Instrumented every stage with metrics: consumer lag, processing latency, watermark delay, and dropped/late event counts. Built dashboards to track pipeline health in real time and set alerting thresholds for lag spikes that indicate the processing layer is falling behind the ingest rate.

Challenges
What Was Hard
⚠️Out-of-order and late-arriving events

Mobile and IoT-style producers don't guarantee delivery order. Naive windowed aggregation either closed windows too early — dropping valid late data — or never closed them, leaking memory.

Implemented bounded-lateness watermarks with allowed lateness windows. Events arriving within the lateness bound update prior aggregations; events beyond it are routed to a dead-letter stream for manual reconciliation.

⚠️State growth under high cardinality

Stateful aggregation across millions of unique keys caused Flink's state backend to balloon, increasing checkpoint duration and risking backpressure on the whole pipeline.

Switched to RocksDB as the state backend for spill-to-disk capability, and added state TTL to expire keys with no recent activity — keeping working memory proportional to active entities, not total history.

⚠️Exactly-once across two sinks

Guaranteeing exactly-once delivery to Kafka is straightforward; guaranteeing it simultaneously to both Redis and Postgres from the same job, without one sink lagging and causing inconsistent reads, was not.

Used Flink's two-phase commit sink protocol for the Postgres path and treated Redis as a best-effort cache layer with short TTLs — accepting at-least-once semantics there since it's not the system of record.

⚠️Backpressure during traffic spikes

Simulated 3x load spikes caused consumer lag to climb faster than auto-scaling could respond, risking a lag spiral where the pipeline never catches up.

Tuned Kafka partition count to allow finer-grained parallel consumption, and added a circuit-breaker pattern that sheds non-critical event types under sustained backpressure to protect core pipeline health.
