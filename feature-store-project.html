Problem Statement
The Challenge
Across multiple ML models, the same underlying signals — income stability, transaction velocity, repayment history — were being recomputed independently by different data scientists, with subtle inconsistencies between training and serving code. This caused training-serving skew, duplicated engineering effort, and made it nearly impossible to know which feature definition was the source of truth.
System Design
Architecture

Raw data is transformed once into feature definitions registered in Feast, materialised to both an offline store for training and an online store for low-latency inference — guaranteeing the same feature value regardless of which path retrieves it.

Build Process
1

Feature Audit & Entity Design

Catalogued every feature already in use across existing models, identifying duplicates and inconsistent definitions of the "same" signal. Defined core entities (user_id, merchant_id) that features would be keyed against, establishing a shared vocabulary across teams before writing any transformation code.

Feature AuditEntity Modelling
2

Transformation Layer

Built SQL and Python transformations for raw-to-feature logic — rolling aggregations (30-day transaction count, 7-day average balance), ratio features (debt-to-income), and stability scores derived from time-series variance. Orchestrated with Airflow DAGs running on a schedule matched to each feature's required freshness.

Apache AirflowSQLPandasWindow Functions
3

Feast Feature Registry

Defined feature views in Feast — declarative specs describing each feature's source table, entity key, data type, and TTL. This registry became the single source of truth: any model requesting "avg_balance_7d" gets the identical definition whether at training time or inference time.

FeastFeature ViewsYAML Definitions
4

Offline Store for Training

Materialised historical feature values into BigQuery as the offline store. Built point-in-time correct joins so that training data reflects exactly what would have been known at each historical timestamp — preventing label leakage from future data sneaking into training features.

BigQueryPoint-in-Time JoinsLeakage Prevention
5

Online Store for Serving

Materialised the latest feature values into Redis as the online store, queried at inference time with sub-50ms latency. Set up incremental materialisation jobs so new feature values land in Redis continuously rather than via slow full-table refreshes.

RedisIncremental MaterialisationLow-Latency Serving
6

Validation & Monitoring

Built automated checks comparing offline and online feature values for the same entity at the same point in time, flagging any drift between the two stores. Added freshness monitoring so stale features (past their TTL) are flagged before they silently degrade model performance.

Challenges
What Was Hard
⚠️Point-in-time correctness for training data

Naively joining features to historical labels by entity ID alone leaked future information into training data — a feature value from after the label date would silently inflate model accuracy and fail in production.

Used Feast's point-in-time join functionality, which matches each label's timestamp against the most recent feature value available before that exact time, not the latest value overall.

⚠️Offline/online store drift

The batch transformation feeding BigQuery and the incremental job feeding Redis occasionally diverged after a schema change in the source table, causing the same feature to return different values depending on which store served it.

Built a nightly reconciliation job comparing a sample of entities across both stores, alerting on any value mismatch above a defined tolerance — caught drift within hours instead of being discovered via model degradation weeks later.

⚠️Feature TTL tuning

Setting TTLs too short caused unnecessary recomputation and Redis churn; too long risked serving stale features as fresh, particularly for fast-moving signals like real-time balance.

Tuned TTLs per feature based on its underlying data volatility rather than a single global default — slow-changing features (income stability) got day-long TTLs, fast-changing ones (balance) got hour-long TTLs.

⚠️Backfilling historical features

Adding a new feature required backfilling years of historical values for training purposes, which on the full raw dataset took hours and risked timing out scheduled jobs.

Partitioned backfill jobs by date range and ran them in parallel via Airflow's dynamic task mapping, cutting backfill time from hours to under twenty minutes for a typical feature.
